# 项目背景文档

## 1. 文档目的

本文档用于沉淀项目的统一背景信息，作为后续设计规范、研发方案、内容生产、数据埋点、上线运维与复盘文档的共同上位说明。

它回答的核心问题是：

- 这个项目为什么要做
- 优先解决什么问题
- 当前阶段做什么、不做什么
- 7 周版本的目标、边界和推进节奏是什么

## 2. 项目概述

本项目拟建设一个 **PWA 形态的扫码即用产品教程与问题解决助手**，面向购买产品后的真实使用场景，帮助用户在无需下载 App、无需注册登录的前提下，快速找到与当前 SKU 和问题相关的教程、提示和解决路径。

项目的首要业务价值不是“做一个功能很多的产品”，而是优先解决以下高频卡点：

- 买回去不会用
- 担心操作不安全
- 担心做错之后没效果

通过缩短用户从“遇到问题”到“找到正确动作”的路径，项目希望降低售后压力，改善使用体验，并进一步带动评价与复购表现。

## 3. 业务背景

从当前项目规划看，目标用户已经不是“泛流量访问者”，而是 **购买后进入使用阶段的真实用户**。他们往往在以下几个时间点最需要帮助：

- 刚收到产品，首次使用时不知道从哪里开始
- 使用过程中出现异响、不适、不会调节等问题
- 对安全性、适用性、正确动作存在疑虑

这类问题有一个共同特征：很多并不一定需要客服一对一介入，而是可以通过更准确的内容组织、更短的解释路径、更贴近场景的引导来解决。如果这部分问题能被前置消化，项目就有机会直接减少售后问题单、退货冲动和差评风险。

## 4. 产品定位

该项目不是传统意义上的内容站，也不是完整的原生 App，而是一个围绕“扫码后立刻解决问题”设计的轻量产品助手。

定位可概括为：

- 入口形态：二维码直达
- 产品形态：PWA
- 使用方式：即开即用，不强依赖安装
- 核心能力：按 SKU 和问题场景提供分步教程、风险提示与问题反馈
- 数据目标：回收教程完成情况、是否解决、未解决原因，形成迭代依据

## 5. 关键决策

当前已明确的关键决策如下：

### 5.1 平台选择

- 默认平台为 PWA，而不是原生 iOS / Android
- 原因是扫码即用更适合低频场景，也更符合当前团队无原生开发能力的现实约束

### 5.2 首要价值判断

- 优先解决“不会用、担心安全、怕做错没效果”导致的使用阻塞
- 优先级高于账号体系、社区互动、复杂训练计划等扩展型能力

### 5.3 MVP策略

- 先做最小闭环，验证真实使用价值和数据回收能力
- 不追求一开始就做成全功能智能助手
- 先围绕 1 到 3 个首批 SKU 打透，而不是大范围铺开

## 6. 项目目标与成功指标

项目当前定义的北极星指标如下：

1. 扫码用户的教程完成率（Tutorial Complete Rate）
2. “是否解决”解决率（Resolved Rate）
3. 相关 SKU 的售后工单下降

其中，前两项更适合作为前期快速验证指标，第三项通常需要更长周期观察。

## 7. 当前明确的约束与边界

7 周版本必须控制范围，避免目标发散。当前已明确的 MVP 不做项包括：

- 账号体系
- 复杂训练计划
- 社区能力
- 全功能 AI 诊断
- 原生 iOS / Android 应用
- 复杂离线策略

这意味着项目第一阶段的重点不是构建完整平台，而是跑通“内容解决问题 + 数据可回收”的最小业务闭环。

## 8. 目标用户与核心场景

根据现有规划，项目优先服务以下几类用户：

- 刚收到产品、0 到 3 天内不会使用的新用户
- 使用过程中遇到问题的用户，例如异响、不适、担心安全
- 已经能顺利使用的用户，用于反向提炼他们的自学路径和最短有效解释

优先解决的不是抽象需求，而是具体的“挣扎时刻”，例如：

- 第一步怎么开始
- 某一步是否安全
- 出现异常是否正常
- 动作/设置是否做对
- 找不到对应型号、零件或入口

## 9. MVP核心闭环

PoC 与 MVP 都围绕同一条核心闭环展开：

`扫码 -> 进入 SKU 页面 -> 进入分步教程 -> 完成教程 -> 提交是否解决 -> 记录未解决原因`

这条链路是整个项目的最小价值单元。后续设计、埋点、内容模板、数据报表都应围绕这条链路展开。

## 10. 阶段性推进框架

### 10.1 第 0 周：成功指标与边界对齐

在正式调研前，先对齐目标指标、MVP边界和首批 SKU 范围，确保后续所有调研和开发都在同一范围内进行。

### 10.2 第 1 周：多维度调研与真需求对齐

这一阶段的核心不是产出很多结论，而是明确三件事：

- 首批要服务哪些 ASIN / SKU
- 用户最核心的 Top1 / Top2 痛点是什么
- 3 分钟内能闭环解决的问题长什么样

阶段 1 的关键输出包括：

- 亚马逊用户反馈数据对齐会纪要
- 问题池 Backlog
- 主战场初判
- JTBD 与伪需求清单
- 核心结论报告
- 核心功能定义与 MVP 范围
- 信息架构与核心页面草图
- 最小埋点与数据回收方案

### 10.3 第 2 周：AI辅助技术路径验证 PoC

这一阶段只验证最小闭环是否能跑通，不追求工程完备性。核心目标包括：

- 扫码可打开
- 能进入 SKU 页面和分步教程页
- 有基础 PWA 能力
- 可以记录关键事件
- 可以导出按 SKU 查看的数据

### 10.4 第 3 到 6 周：MVP开发

这一阶段的目标是把 PoC 打磨成一个可用、可扩展、可运营的 MVP，并完成首批内容、测试、基础看板与运营手册。

### 10.5 第 7 周：小流量上线与数据复盘

通过包裹卡片、说明书贴纸、售后自动回复等渠道做小流量投放，用真实用户行为验证漏斗，并输出下一轮迭代优先级。

## 11. MVP功能范围

### 11.1 必须有

- SKU 落地页
- 问题快捷入口
- 分步教程
- “是否解决”反馈
- 未解决原因采集
- 最低限度的数据统计或导出能力

### 11.2 应该有

- 搜索
- 多语言支持，至少英文
- 轻量内容管理，最好支持表格或 JSON 驱动

### 11.3 可以后置

- 零件匹配图库
- 训练记录或成就体系
- 受控知识库问答

## 12. 数据回收原则

项目从第一版开始就必须具备基础数据回收能力，否则无法判断内容是否真的解决问题。

当前最小埋点建议包括：

- `scan_open`
- `sku_view`
- `tutorial_start`
- `step_complete`
- `tutorial_complete`
- `resolved`
- `unresolved_reason`

无论最终使用 PostHog、GA4 还是 Google Sheet，原则都是先保证关键闭环数据可用，再考虑更复杂的分析能力。

## 13. 内容建设原则

本项目的核心竞争力很大程度上来自内容组织，而不是功能堆叠。教程内容应尽量统一结构，便于复用、翻译和扩展。

建议统一字段至少包括：

- 适用人群
- 风险提示
- 使用前准备
- 分步说明
- 常见错误
- 何时停止
- FAQ

这也是后续内容模板、CMS/表格结构、设计规范的重要输入。

## 14. 合规与体验原则

考虑到项目可能涉及安全、健康、身体不适等敏感场景，内容和产品表达应遵循以下原则：

- 安全提示前置
- 避免不当医疗承诺或高风险措辞
- 对孕期、疼痛、不适等场景提供谨慎表述
- 明确隐私声明，说明数据采集范围和用途
- 在弱网、扫码、首次打开等场景优先保证可用性

## 15. 风险与应对

当前最关键的项目风险包括：

### 15.1 调研结论分散，导致MVP发散

应对方式：

- 阶段 1 必须收敛出主战场、Top1 / Top2 痛点和不做清单
- 未达成收敛前，不进入 PoC

### 15.2 PoC投入过深，影响整体节奏

应对方式：

- PoC 只验证最小闭环和数据回收
- 非关键能力全部后置

### 15.3 没有真实数据，无法驱动迭代

应对方式：

- 从 PoC 开始接入最小埋点
- “是否解决”和“未解决原因”必须进入 MVP

## 17. 一句话总结

这是一个以 **扫码即用、快速解决问题、可回收真实反馈** 为核心的 PWA 项目。它的成功不取决于功能是否复杂，而取决于是否能围绕少量核心 SKU，在最短路径内帮用户解决真实使用问题，并把结果沉淀为可迭代的数据。
