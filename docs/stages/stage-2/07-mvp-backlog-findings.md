# 阶段二：MVP Backlog Findings

## 1. 结论

阶段二 PoC 已验证“安装引导 + 反馈闭环 + 基础事件回收”可以成立，但离 MVP 还有几项必须收敛的问题。

## 2. Must Fix Before MVP

| 问题 | 影响 | 建议动作 | 优先级 |
|---|---|---|---|
| Step 1 仍然是最高风险步骤，当前仅靠静态对比说明 | 无法充分解决“压扣打滑”这一核心退货原因 | 升级为明确的错误/正确安装动图或短视频，并强化走带方向提示 | P0 |
| 埋点仍使用 `localStorage` 本地日志兜底 | 阶段三无法形成真实分析面板与复盘数据 | 接入 PostHog 或 GA4，并保留当前事件命名 | P0 |
| 内容仍在代码内，尚未形成可扩展的 ASIN 数据结构 | 后续 SKU 扩展成本高，文案改动会牵动代码 | 抽离为 JSON 或内容配置文件，按 ASIN 建模 | P0 |
| 支持与完成状态仍以 `alert` 方式表达 | 体验不完整，也不利于后续测量与留存 | 改为正式的完成页、支持页或内嵌状态卡片 | P1 |
| 移动端只完成了浏览器模拟验证 | 康复场景真实使用风险较高，仍缺少设备侧确认 | 补充 iPhone Safari 和 Android Chrome 的实机走查记录 | P1 |

## 3. Should Improve for MVP

| 问题 | 影响 | 建议动作 | 优先级 |
|---|---|---|---|
| 当前页面通过 hash 状态切换，深链接和恢复能力有限 | 后续分享、回看和多页面扩展会受限 | 评估升级为真实路由结构 | P2 |
| Step 2 的“安全证明”仍偏 PoC 表达 | 康复用户对安全的焦虑可能仍无法充分缓解 | 增加缝线微距图、承重说明和异常停用提示 | P2 |
| Step 3 的示意图仍是抽象图形 | 对“合页侧安装”的新手教育力有限 | 加入更直观的门板方向示意或真人图示 | P2 |
| 当前验证脚本只覆盖一条完整链路和一条流失链路 | 后续回归覆盖不足 | 增加 resolved=yes 和 support fallback 的自动验证 | P2 |

## 4. Later / Nice to Have

| 机会点 | 预期价值 | 备注 |
|---|---|---|
| 真人语音旁白 | 降低独居用户和康复用户理解成本 | 对应阶段一 Should 范围 |
| 康复动作示范入口 | 建立专业信任，提高二次使用率 | 可作为阶段三或四扩展 |
| 客服转接与帮助入口 | 承接 `door_not_fit` 等高风险场景 | MVP 可先给轻量联系方式 |

## 5. 推荐带入阶段三的重点

1. 优先把 Step 1 做成真正能降低打滑的核心功能页，而不是只做视觉说明
2. 保留当前事件命名，尽快迁移到正式分析平台
3. 将内容模型独立出来，为多 SKU 和文案迭代做准备
4. 把 PoC 中的 `alert` 替换为正式页面状态，补齐真实用户体验
